# Score Source Type Detection Design

## Context

The app already has one explicit format check for MuseScore SVG imports, but it does not maintain a unified source classification across supported notation exporters.

Current behavior is split across two different mechanisms:

- MuseScore is identified with a dedicated helper based on `<desc>` content and semantic classes.
- Dorico and Sibelius imports are handled indirectly through font sniffing and format-specific compatibility paths.

That leaves two gaps:

1. There is no single source of truth for the imported score type.
2. The detected source is not exposed in the UI, so users cannot tell how the app classified a file.

The user has provided three exports of the same score as representative samples:

- `Dorico Type.svg`
- `Musescore Type.svg`
- `Sibelius Type.svg`

Those samples show stable high-signal markers:

- Dorico: `<desc>Dorico</desc>`
- MuseScore: `<desc>Generated by MuseScore Studio ...</desc>` and semantic classes such as `Clef`, `KeySig`, `TimeSig`, `Bracket`, `BarLine`
- Sibelius: Qt-style export plus Sibelius font families such as `Opus Std` and `Opus Special Std`

## Goal

Add a centralized score-source classifier that returns one of:

- `MuseScore`
- `Dorico`
- `Sibelius`
- `Unknown`

Then surface that result in the UI immediately after SVG import while preserving all current detection behavior.

## Options Considered

### Option 1: Keep the current MuseScore helper and add a separate UI-only classifier

Pros:

- Smallest immediate change.
- Keeps existing MuseScore branches untouched.

Cons:

- Two overlapping source-detection paths would drift over time.
- Future vendor-specific behavior would still have no single source of truth.

### Option 2: Centralize source detection and make MuseScore use it

Pros:

- One canonical classification result for UI and runtime logic.
- Minimal conceptual overhead.
- Easy to extend later if more exporters need explicit handling.

Cons:

- Requires touching a few existing MuseScore call sites.

### Option 3: Build a full vendor abstraction layer now

Pros:

- Cleanest long-term abstraction on paper.

Cons:

- Overbuilt for the current need.
- Larger regression surface with no immediate user benefit.

## Decision

Use Option 2.

Introduce one centralized helper that classifies the imported SVG source, then derive the current MuseScore branch from that result instead of maintaining a separate detection mechanism.

## Design

### Centralized classifier

Add a helper in `scripts/app.js`:

- `detectScoreSourceType(svgRoot, svgText = "")`

Return values:

- `MuseScore`
- `Dorico`
- `Sibelius`
- `Unknown`

Detection order should prefer the strongest explicit signals first:

1. MuseScore
2. Dorico
3. Sibelius
4. Unknown

This order avoids downgrading a file with explicit MuseScore metadata just because it also contains recognizable music-font names.

### Vendor rules

#### MuseScore

Treat the file as MuseScore when either of these is true:

- `<desc>` contains `Generated by MuseScore`
- the SVG contains known MuseScore semantic classes such as `Clef`, `KeySig`, `TimeSig`, `Bracket`, or `BarLine`

This preserves the existing class-first behavior for MuseScore imports.

#### Dorico

Treat the file as Dorico when the metadata clearly identifies it as Dorico, such as:

- `<desc>` containing `Dorico`

The provided Dorico sample uses this exact marker, so this rule is both simple and strong.

#### Sibelius

Treat the file as Sibelius when the file lacks stronger MuseScore or Dorico markers but contains strong Sibelius-family font/export signals, such as:

- `Opus Std`
- `Opus Special Std`
- `Opus Text Std`
- `Helsinki`
- `Inkpen2`

The classifier should inspect both the raw SVG text and the DOM so it can detect these signals even when they appear inside nested groups or style fragments.

#### Unknown

Return `Unknown` when no strong vendor signal is present.

This fallback is important because the app already supports partial compatibility through generic geometric and font-based logic, and we should not force a false classification.

### Runtime state

Add a runtime state field to store the current source type, initialized to `Unknown`.

During `processSvgContent(...)`:

- reset the state to `Unknown` before parsing
- classify the newly imported SVG as soon as the root is available
- persist the result in state and any existing compatibility booleans derived from it

`isMuseScoreSvg()` should become a thin wrapper over the centralized result or centralized helper so existing MuseScore-specific logic keeps working.

### UI display

Expose the current score source in the `SOURCES` card because the value belongs to the imported SVG itself.

Add a compact read-only info row with text similar to:

- `谱子类型：-` before import
- `谱子类型：MuseScore` after importing a MuseScore SVG

The display should update:

- on successful SVG import
- on failed import reset
- when the app clears or replaces the current SVG

### Non-Goals

- Changing how music-font auto-detection works
- Refactoring all vendor-specific logic into separate modules
- Adding new vendor-specific detection behavior beyond the four requested classifications
- Fixing unrelated pre-existing Dorico time-signature regressions

## Testing

Add focused smoke coverage that proves:

1. the classifier returns `Dorico`, `MuseScore`, `Sibelius`, and `Unknown` for dedicated fixtures
2. importing those fixtures updates the UI label with the expected source type
3. existing MuseScore class-first behavior still passes through the centralized source-type path

The new tests should be independent of the currently failing unrelated Dorico time-signature regression so this feature can be verified on its own merits.
