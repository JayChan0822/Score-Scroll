# MuseScore Class-First Detection Design

## Context

MuseScore Studio exports SVGs differently from Sibelius and Dorico in a way that breaks the current opening-symbol detection chain:

- The file may contain no `<text>` nodes for notation symbols.
- The file may contain no `font-family` attributes, so music-font sniffing cannot infer the active notation font.
- Semantic classes are already present on the exported geometry, including `Clef`, `KeySig`, `TimeSig`, `Bracket`, `BarLine`, `Accidental`, `Rest`, `Dynamic`, and `Note`.

The current detector still assumes one of two formats:

1. Text glyphs that can be recognized by character content.
2. Path glyphs that can be recognized by a font-specific path signature registry.

That assumption fails for MuseScore opening symbols because the exported paths are semantically labeled but do not match the current path-signature registry.

## Goal

For MuseScore-exported SVGs, prefer semantic `class` names over font sniffing and path-signature guessing, while preserving the existing fallback chain for Sibelius, Dorico, and other SVGs.

## Options Considered

### Option 1: Keep signature-first detection and expand the registry

Pros:

- Minimal structural change.
- Reuses the current detection pipeline.

Cons:

- Requires adding many MuseScore-specific path signatures.
- Still ignores the semantic classes already present in the SVG.
- Will remain brittle across MuseScore versions or alternate engraving fonts.

### Option 2: Add MuseScore-specific class-first detection, keep existing logic as fallback

Pros:

- Uses the strongest signal already present in MuseScore SVG output.
- Small, targeted change that does not disturb the existing signature/text flows.
- Scales naturally to other MuseScore semantic classes later.

Cons:

- Introduces a format-specific branch in the detection pipeline.
- Requires a reliable MuseScore export check.

### Option 3: Build a generic class-alias framework for all vendors now

Pros:

- Long-term extensibility.
- Clean abstraction if multiple exporters converge on semantic classes.

Cons:

- Overbuilt for the current bug.
- Higher regression risk and larger test surface.

## Decision

Use Option 2.

The initial implementation will:

- Detect MuseScore exports from SVG metadata and semantic class presence.
- Prefer semantic class names for opening `Clef`, `KeySig`, `TimeSig`, and `Bracket` detection.
- Keep the existing text/signature-based logic as fallback for non-MuseScore files and for any MuseScore elements without supported classes.

## Design

### MuseScore detection

Add a helper that treats an SVG as MuseScore-exported when at least one of the following is true:

- The `<desc>` contains `Generated by MuseScore`.
- The document contains known MuseScore semantic classes such as `Clef`, `KeySig`, `TimeSig`, `Bracket`, or `BarLine`.

The helper should be conservative and cheap to evaluate.

### Opening symbol classification order

For the opening detectors only, use this order:

1. MuseScore semantic classes when the file is identified as MuseScore.
2. Existing text-based detection.
3. Existing path-signature detection.

This keeps the new behavior format-scoped rather than globally changing precedence for every SVG.

### Bracket handling

MuseScore exports the piano/choir opening bracket as `path class="Bracket"`, while the current bracket detector only scans `polyline` and `line`.

For MuseScore files:

- Mark `path.Bracket` as `highlight-brace` directly.
- Leave geometric bracket detection in place as fallback for non-MuseScore SVGs and for line/polyline bracket structures.

### Clef handling

For MuseScore files:

- Mark `path.Clef` as `highlight-clef` directly.
- Continue to populate the main clef list so later anchor logic still works.

### Key signature handling

For MuseScore files:

- Mark `path.KeySig` as `highlight-keysig` directly when it is not obviously left of the system start.
- Preserve the accidental contagion safeguards already added earlier.

### Time signature handling

For MuseScore files:

- Collect `path.TimeSig` as first-class candidates.
- Validate them with the same spatial rules already used for text-based time signatures so false positives remain bounded.

## Non-Goals

- Replacing the entire symbol detector with a vendor-specific semantic parser.
- Adding class-first handling for every MuseScore class in this change.
- Refactoring the music-font registry.

## Testing

Add a fixture derived from the provided MuseScore file and a regression test that verifies:

- Opening `Bracket` path is classified as `highlight-brace`.
- Opening `Clef` paths are classified as `highlight-clef`.
- Opening `KeySig` paths are classified as `highlight-keysig`.
- Opening `TimeSig` paths are classified as `highlight-timesig`.

The regression should fail on the current code and pass only after the class-first compatibility layer is added.
